# Avatar-State Research System

Read when:
- adding new Reddit raw exports
- refreshing the beachhead avatar workspace
- defining a new state cluster or adjacent sub-avatar
- deciding what to read first: broad slice vs priority slice

## Goal

Run avatar research as a scalable, single-avatar, multi-state system instead of one-off manual digging.

This repo now supports:
- incremental corpus updates from new raw Reddit drops
- canonical thread reconstruction
- orphan-thread preservation
- quote candidate refresh
- avatar/state-cluster slicing
- priority subset generation for frontline DR reading
- pack generation for both broad and priority slices
- long-form report rendering to markdown + HTML + PDF

## Core Model

Do not model the market as broad `tinnitus`.

Model it as recurring states:
- nighttime spike
- jaw tension / clenching
- quiet-room amplification
- lying-down fear
- night-guard confusion
- normal-hearing-but-miserable

This keeps the research anchored to a believable symptom loop and gives copy/mechanism work a tighter lane than generic tinnitus messaging.

## Update Flow

When new raw Reddit files arrive:

1. Drop them under `data/raw/<subreddit>/`
2. Keep raw files immutable
3. Run:

```bash
python scripts/build_avatar_workspace.py \
  --config configs/avatar_states/jaw_clenching_nighttime_spike.json \
  --seed-csv /absolute/path/to/manual_source_log.csv
```

What happens:
- raw pair discovery under `data/raw/`
- registry check at `data/intermediate/ingest_registry.json`
- canonical corpus refresh only when new/changed pairs exist
- quote candidates refresh only when corpus changed or quote output is missing
- avatar slice rebuild
- priority slice rebuild
- packs rebuild

## Broad vs Priority

### Broad slice

Use for:
- discovery
- pattern counting
- new angle mining
- adjacent-cluster detection

It is intentionally wider.

### Priority slice

Use for:
- first-pass reading
- thread snapshots
- quote mining for copy
- mechanism-angle work
- ad/lander strategy

It is narrower and closer to the frontline beachhead.

## Adjacent Cluster Logic

Do not hard-freeze the system around one avatar forever.

Use the broad slice and report outputs to watch for adjacent clusters emerging, for example:
- paper-normal still miserable
- midlife sleep disruption
- bite-change / dental-trigger cases
- nervous-system panic loop variants

When an adjacent cluster becomes large enough:
- create a new config under `configs/avatar_states/`
- rerun the same workspace build
- compare outputs without changing the ingest layer

## Design Rules

- canonical ingest must be deterministic
- raw files stay immutable
- research claims should anchor to thread evidence
- broad slice for discovery, priority slice for action
- new sub-avatars should be added as configs, not hacked into one giant persona
